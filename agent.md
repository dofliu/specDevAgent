# specDevAgent 行動指南

本文件定義 AI 代理人在本專案框架內執行任務時的行為規範。代理人需以
文件驅動的方式工作，確保每一次迭代都具備可追溯性與可驗證性。

## 任務循環
1. **PLAN** — 研讀 `project.md`、`todo.md` 與最新的 `development.log`，釐清任務目標。
2. **CHANGES** — 在分支上依需求修改程式碼與文件，必要時新增 ADR。
3. **TEST** — 執行自動化測試與靜態檢查，蒐集輸出紀錄。
4. **GIT** — 以 Conventional Commits 撰寫提交，摘要需能對應 `todo.md` 任務。
5. **LOG** — 更新 `development.log`，記錄此次變更的原因、測試結果與下一步。
6. **DoD** — 確保文件、程式碼與測試皆已同步且通過驗收，再送出 PR。

## 文件維護守則
- `project.md`：保持專案目標、成功指標與利害關係人資訊最新。
- `todo.md`：新增或更新任務條目，標示狀態（未開始 / 進行中 / 已完成）。
- `development.log`：每次提交需新增區塊，紀錄摘要、測試指令與結果。
- `docs/decisions/*.md`：重要架構或流程決策需建立 ADR，並附上狀態。

## 測試與品質
- 變更應附帶至少一項自動化測試或說明既有測試涵蓋該情境。
- CLI 或腳本需支援 `--help` 說明，並在失敗時提供可判讀訊息。
- 不得跳過測試步驟；若受限於環境需在 `development.log` 註記原因。

## 沟通與協作
- PR 訊息需包含變更摘要、測試指令與結果。
- 若遇到阻塞議題，應在 `todo.md` 新增風險或疑問並指派後續動作。
- 文件與程式碼差異超過一個迭代時，須優先回補文件以維持可靠性。

遵循上述規範可確保代理人在 specDevAgent 模板下進行的每一次交付皆具備高可靠性與可審核性。
